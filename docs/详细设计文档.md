## 1. 系统概述

### 1.1 项目定位

无缝钢管（Casing / Tubing）与筛管（Screen Pipe）进销存一体化 Web 管理系统，覆盖管材从入厂到出厂的全流程数据追踪，基于 API 5CT / ISO 11960 标准。

### 1.2 核心能力

| 能力 | 说明 |
|------|------|
| 管材全生命周期管理 | 从采购入库到销售出库的完整追踪 |
| 进销存一体化 | 采购、库存、销售的联动管理 |
| 质量追溯 | 炉批号/管材编号维度的质检追溯 |
| 多用户 RBAC | 仓库管理员、质检员、销售/采购人员、管理员四角色 |
| 国际化 | 中英文界面 + 公制/英制单位切换 |

| **异步运行时** | Tokio | 1.x | Rust 异步运行时标准 |
| **参数校验** | Validator | 0.18+ | 基于 derive 宏的结构体校验 |
| **日志** | Tracing + Tracing-Subscriber | — | 结构化日志，与 Tokio 生态集成 |
| **文件上传** | Axum multipart + tokio-util | — | 处理质检文件上传 |
| **国际化** | rust-i18n / Fluent | — | 中英文消息模板 |
| **导出/导入** | calamine (读) + xlsxwriter (写) | — | Excel 文件读写 |

**RESTful 前后端分离**：后端提供 JSON REST API，前端通过 HTTP 调用。

## 5. 数据库详细设计

### 5.1 设计原则

- **SQLite WAL 模式**：通过 `PRAGMA journal_mode=WAL;` 启用，支持读-写并发
- **不使用外键约束**：SQLite 外键性能开销大，通过应用层保证引用完整性
- **索引策略**：查询频繁的字段建索引，组合查询建联合索引
- **时间字段**：统一使用 ISO 8601 文本格式存储 (`TEXT`)，兼容性好
- **枚举字段**：使用 `TEXT` 类型存储，便于可读性和扩展
- **软删除**：关键数据表增加 `deleted_at` 字段，不物理删除

### 5.2 数据库初始化配置

```sql
-- 启用 WAL 模式
PRAGMA journal_mode = WAL;
-- 启用外键（应用层控制，但仍开启校验）
PRAGMA foreign_keys = ON;
-- 设置 busy timeout
PRAGMA busy_timeout = 5000;
-- 同步模式：NORMAL 兼顾性能与安全
PRAGMA synchronous = NORMAL;
-- 缓存大小：64MB
PRAGMA cache_size = -64000;
-- 临时存储：内存
PRAGMA temp_store = MEMORY;
```

| `422` | 请求体校验失败（Validator） |
| `500` | 服务器内部错误 |

### 8.2 错误码定义

```rust
pub enum AppErrorCode {
    // 通用 (100xx)
    InternalError,          // 10001
    ValidationError,        // 10002
    NotFound,               // 10003

    // 认证 (110xx)
    Unauthorized,           // 11001
    TokenExpired,           // 11002
    Forbidden,              // 11003

    // 管材 (120xx)
    PipeNotFound,           // 12001
    PipeNumberDuplicate,    // 12002
    PipeStatusConflict,     // 12003 (如已出库的管材不能删除)

    // 库存 (130xx)
    InsufficientStock,      // 13001
    LocationFull,           // 13002
    LocationNotFound,       // 13003

    // 订单 (140xx)
    OrderCannotModify,      // 14001 (已审核订单不能修改)
    OrderNotFound,          // 14002
    OrderNotApproved,       // 14003 (订单未审核，不能用于入库/出库)

    // 库存 (130xx续)
    InboundOrderIdRequired,         // 13004 (采购入库必须关联采购订单)
    OutboundOrderIdRequired,        // 13005 (销售出库必须关联销售订单)
    InboundNotApprovedYet,          // 13006 (入库单尚未审批)
    OutboundNotApprovedYet,         // 13007 (出库单尚未审批)
    InboundAlreadyApproved,         // 13008 (入库单已审批)
    OutboundAlreadyApproved,        // 13009 (出库单已审批)
    InboundItemMismatch,            // 13010 (入库管材与采购订单不匹配)
    OutboundItemMismatch,           // 13011 (出库管材与销售订单不匹配)
    InboundApprovalNotAllowed,      // 13012 (无权审批非采购入库)
    OutboundApprovalNotAllowed,     // 13013 (无权审批非销售出库)

    // 导入导出 (150xx)
    ImportFileParseError,   // 15001
    ImportValidationError,  // 15002
}
```

## 9. 非功能设计

### 9.1 性能设计

| 指标 | 方案 |
|------|------|
| **查询响应 ≤ 2s（10 万条）** | 合理的索引设计 + 分页查询 + 避免 N+1 查询 |
| **导入 10 万条 ≤ 60s** | 批量插入（SQLx 的 `execute_many`），事务分批提交（每批 1000 条） |
| **并发 20+ 人** | SQLite WAL 模式允许读写并发；写入序列化通过 `busy_timeout` |
| **大量数据查询** | 报表等复杂聚合查询使用缓存（后续可引入 SQLite 物化视图或应用层缓存） |

**SQLite 并发注意事项**：

```
1. 使用 WAL 模式（默认配置），读取不阻塞写入
2. 写入操作通过 Tokio 的 Mutex 序列化，避免 SQLITE_BUSY
3. 连接池：r2d2_sqlite 或 deadpool-sqlite，最大连接数 = 5
   （WAL 模式下，多个 reader 可同时工作）
4. 长时间运行的写入事务拆分为小批量事务
5. 定期执行 PRAGMA wal_checkpoint(TRUNCATE) 限制 WAL 文件大小
```

```
    ├── messages.ftl
    ├── errors.ftl
    └── api_5ct.ftl
```

- 错误消息、校验提示、状态标签在服务端根据 `Accept-Language` Header 或用户偏好返回对应语言
- 用户 `language_pref` 字段存储语言偏好

**前端**：

- 切换语言时更新 `language_pref` 并刷新页面内容

### 11.2 单位切换方案

**策略**：内部存储统一使用**英制单位**（API 5CT 标准单位），API 返回/接收时根据用户偏好进行单位转换。

| 参数 | 内部存储（英制） | 公制显示 | 转换公式 |
|------|----------------|---------|---------|
| 外径 (OD) | inch | mm | mm = in × 25.4 |
| 壁厚 (WT) | inch | mm | mm = in × 25.4 |
| 长度 | foot | m | m = ft × 0.3048 |
| 单位重量 | lb/ft | kg/m | kg/m = lb/ft × 1.48816 |
| 屈服强度 | psi | MPa | MPa = psi × 0.00689476 |

**API 设计**：

```json
// 请求中可选指定单位
POST /api/v1/seamless-pipes?unit_system=metric

// 响应中增加 unit 标记
{
  "data": {
    "od": 114.3,          // 公制值
    "od_unit": "mm",
    "wt": 6.35,
    "wt_unit": "mm",
    "length": 12.19,
    "length_unit": "m"
  }
}
```

**实现**：

- 在 DTO 层定义 `Measurement<T>` 包装类型，携带单位信息
- 在 `AuthService` 中从用户配置获取单位偏好
- 在 Handler 层，通过中间件提取用户单位偏好，传递给 DTO 转换
- 前端也可以做前端单位转换（用户侧），由前端根据 `unit_system` 偏好自行转换显示

## 附录 A：关键决策记录 (ADR)

### ADR-001：选择单体架构而非微服务

| 项目 | 内容 |
|------|------|
| **背景** | 项目规模中等，团队规模小 |
| **决策** | 采用模块化单体架构，内部按领域分包，不拆分为独立微服务 |
| **理由** | 降低运维复杂度，SQLite 单数据库不支持微服务；后续如需拆分可按模块边界提取为独立服务 |
| **后果** | 需要严格的模块边界设计，防止模块间耦合 |

### ADR-002：选择 SQLite 而非 PostgreSQL

| 项目 | 内容 |
|------|------|
| **背景** | 数据规模数十万条，并发 20+ 人 |
| **决策** | SQLite WAL 模式，配合连接池和写入序列化 |
| **理由** | 零配置，文件级部署，适合此规模；估算 SQLite 可支撑百万级读和中等写并发 |
| **后果** | 后续如增长到更大规模可迁移至 PostgreSQL（SQLx 对两者兼容性好） |

### ADR-003：英制单位作为内部存储标准

| 项目 | 内容 |
|------|------|
| **背景** | API 5CT 标准使用英制单位，国内用户习惯公制 |
| **决策** | 数据库统一存储英制值，API 层根据用户偏好转换 |
| **理由** | 避免数据精度损失（公制↔英制反复转换会累积误差），与标准保持一致 |
| **后果** | 所有内部计算使用英制，API 输入/输出时做单位转换 |

### ADR-004：无缝钢管与筛管分表存储

| 项目 | 内容 |
|------|------|
| **背景** | 两种管材虽有共同属性，但筛管有基管参数、过滤精度等特有字段 |
| **决策** | 两个独立表，不共用表结构 |
| **理由** | 字段差异大（筛管有特有字段且基管本身就是无缝钢管参数），联合查询场景少；分表更清晰 |
| **后果** | 跨类型搜索需要 `UNION` 查询，或两次独立查询后合并 |
