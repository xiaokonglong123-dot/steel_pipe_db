## 1. System Overview

### 1.1 Project Scope

A full-cycle Web management system for seamless steel pipe (Casing / Tubing) and Screen Pipe — covering procurement, inventory, sales, QC, and traceability from factory gate to shipping dock. Compliant with API 5CT / ISO 11960 standards.

### 1.2 Core Capabilities

| Capability | Description |
|-----------|-------------|
| Full lifecycle pipe tracking | Complete traceability from purchase inbound to sales outbound |
| Integrated procurement-inventory-sales | Linked management across purchasing, stock, and sales |
| Quality traceability | Heat number / pipe number dimension QC traceability |
| Multi-user RBAC | 4 roles: warehouse admin, QC inspector, sales/purchasing, admin |
| Internationalization | Chinese/English UI + metric/imperial unit switching |

## 2. Tech Stack Decisions

### 2.1 Backend Technology

| Layer | Choice | Version | Rationale |
|-------|--------|---------|-----------|
| **Web Framework** | Axum | 0.8+ | Most mainstream Rust async web framework, Tower middleware ecosystem, active community |
| **ORM / SQL** | SQLx | 0.8+ | Compile-time SQL checking, non-intrusive, native SQLite support |
| **Database** | SQLite (WAL mode) | 3.46+ | Zero config, file-level database; WAL mode supports read-write concurrency |
| **Serialization** | Serde + Serde JSON | 1.x | Rust standard serialization framework |
| **Auth** | Axum + JWT (jsonwebtoken) | — | Stateless auth, works well for SPA separation |
| **Password Hashing** | Argon2 | — | OWASP-recommended password hashing algorithm |
| **Async Runtime** | Tokio | 1.x | Rust async runtime standard |
| **Validation** | Validator | 0.19+ | Derive-macro based struct validation |
| **Logging** | Tracing + Tracing-Subscriber | — | Structured logging, integrates with Tokio |
| **File Upload** | Axum multipart + tokio-util | — | QC file upload handling |
| **Internationalization** | rust-i18n / Fluent | — | Chinese/English message templates |
| **Export/Import** | calamine (read) + xlsxwriter (write) | — | Excel file read/write |

### 2.2 Frontend Technology

The frontend is a React 19 SPA with the following architecture:

| Layer | Choice | Notes |
|-------|--------|-------|
| **UI Framework** | React 19 | Latest stable |
| **Routing** | react-router-dom v7 | createBrowserRouter with protected routes |
| **Client State** | Zustand 5 | Auth store, app store, unit conversion store |
| **Server State** | TanStack Query 5 | 2-min staleTime, 5-min gcTime, retry: 1 |
| **UI Component Library** | Ant Design 5 | Enterprise-grade, great for admin dashboards |
| **Internationalization** | react-i18next | Per-feature namespaces, zh-CN primary + en |
| **HTTP Client** | Axios | Instance at `/api/v1`, auto Bearer token |
| **Validation** | Zod 3 | Runtime response validation via `validateResponse.ts` |
| **Type Safety** | TypeScript 5 | Strict mode, noUnusedLocals, noUnusedParameters enforced |
| **Build** | Vite 6 | Dev proxy `/api/*` → `localhost:3000` |

**Feature modules (11)**: auth, pipes, inventory, suppliers, customers, purchases, sales, quality, contracts, reports, labels. Each feature has `api/`, `pages/`, `types/` subdirectories.

### 2.3 Architecture Style

**RESTful front-back separation**: Backend serves JSON REST API, frontend consumes via HTTP.

**Modular monolith** (not microservices): Given project scale and team size, we use a modular monolith with domain-based layering internally.

## 5. Database Design

### 5.1 Design Principles

- **SQLite WAL mode**: Enabled via `PRAGMA journal_mode=WAL;` for read-write concurrency
- **No foreign key constraints**: SQLite FK performance overhead is significant; referential integrity enforced at the application layer
- **Indexing strategy**: Index frequently queried fields; composite indexes for combined queries
- **Timestamp format**: ISO 8601 text format (`TEXT`) for compatibility
- **Enum fields**: Stored as `TEXT` for readability and extensibility
- **Soft delete**: Key tables include `deleted_at` column; no physical deletion

### 5.2 Database Initialization

```sql
-- Enable WAL mode
PRAGMA journal_mode = WAL;
-- Enable foreign keys (app-layer control, but keep validation on)
PRAGMA foreign_keys = ON;
-- Set busy timeout
PRAGMA busy_timeout = 5000;
-- Sync mode: NORMAL balances performance and safety
PRAGMA synchronous = NORMAL;
-- Cache size: 64MB
PRAGMA cache_size = -64000;
-- Temp storage: memory
PRAGMA temp_store = MEMORY;
```

## 6. REST API Design

### 6.1 API Base Convention

**Base path**: `/api/v1`

**Unified response format**:

```json
// Success response
{
  "success": true,
  "data": { ... },
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 100
  },
  "request_id": "req_xxxxx"
}

// Error response
{
  "success": false,
  "error": {
    "code": "PIPE_NOT_FOUND",
    "message": "Pipe not found",
    "details": { "pipe_id": 123 }
  },
  "request_id": "req_xxxxx"
}
```

**Auth method**: `Authorization: Bearer <jwt_token>`

**Pagination params**:

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `page` | integer | 1 | Page number |
| `page_size` | integer | 20 | Items per page (max 100) |

### 6.2 Pipe Management API

#### 6.2.1 Seamless Pipe

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/v1/seamless-pipes` | List query (supports combined filters) |
| `POST` | `/api/v1/seamless-pipes` | Create seamless pipe |
| `GET` | `/api/v1/seamless-pipes/{id}` | Get details |
| `PUT` | `/api/v1/seamless-pipes/{id}` | Full update |
| `DELETE` | `/api/v1/seamless-pipes/{id}` | Delete (soft delete) |

| `od_min` / `od_max` | number | OD range |
| `wt_min` / `wt_max` | number | WT range |
| `status` | string | Stock status |
| `location_id` | integer | Location ID |
| `manufacturer` | string | Manufacturer |
| `sort_by` | string | Sort field (default `created_at`; whitelist only: `created_at`, `pipe_number`, `grade`, `od`, `wt`, `status`) |
| `sort_order` | string | asc / desc |

**POST /api/v1/inbound-records request body — Purchase inbound**:

```json
{
  "inbound_type": "purchase",
  "inbound_date": "2026-05-19",
  "order_id": 1,
  "supplier_id": 1,
  "operator_id": 1,
```

```json
{
  "inbound_type": "production",
  "inbound_date": "2026-05-19",
  "supplier_id": 1,
  "operator_id": 1,
  "remark": "Production return (finished pipe back to stock)",
  "pipes": [
    { "pipe_type": "seamless", "pipe_id": 100 }
  ]
}
```

> **Approval flow**: When inbound_type='production' or 'return', status is 'pending' after creation. Stock updates only happen after approval.
> **Batch inbound**: If `pipes` is empty and `batch_create` is provided, system auto-creates N pipes of the same spec and records inbound. Batch inbound only supports inbound_type='purchase'.
> **Batch inbound endpoint**:

## 9. Non-Functional Design

### 9.1 Performance Design

| Metric | Approach |
|--------|----------|
| **Query response ≤ 2s (100k rows)** | Proper index design + pagination + avoid N+1 queries |
| **Import 100k rows ≤ 60s** | Batch insert (SQLx `execute_many`), transactional batch commits (1k per batch) |
| **Concurrent 20+ users** | SQLite WAL mode allows read-write concurrency; write serialization via `busy_timeout` |
| **Large data queries** | Cache for complex aggregation queries (SQLite materialized views or app-layer caching in the future) |

**SQLite concurrency notes**:

```
1. Use WAL mode (default config), reads don't block writes
2. Write operations serialized via Tokio Mutex to avoid SQLITE_BUSY
3. Connection pool: r2d2_sqlite or deadpool-sqlite, max connections = 5
   (WAL mode allows multiple concurrent readers)
4. Long-running write transactions split into small batches
5. Periodic PRAGMA wal_checkpoint(TRUNCATE) to limit WAL file size
```

### 11.2 Unit Switching Strategy

**Approach**: Internal storage uses **imperial units** (API 5CT standard). API returns/accepts based on user preference with conversion.

| Parameter | Internal (Imperial) | Metric Display | Conversion |
|-----------|-------------------|----------------|------------|
| Outer diameter (OD) | inch | mm | mm = in × 25.4 |
| Wall thickness (WT) | inch | mm | mm = in × 25.4 |
| Length | foot | m | m = ft × 0.3048 |
| Unit weight | lb/ft | kg/m | kg/m = lb/ft × 1.48816 |
| Yield strength | psi | MPa | MPa = psi × 0.00689476 |

**API Design**:

```json
// Optionally specify units in request
POST /api/v1/seamless-pipes?unit_system=metric

// Response includes unit markers
{
  "data": {
    "od": 114.3,          // Metric value
    "od_unit": "mm",
    "wt": 6.35,
    "wt_unit": "mm",
    "length": 12.19,
    "length_unit": "m"
  }
}
```

**Implementation**:

- `Measurement<T>` wrapper type in DTO layer, carries unit info
- AuthService retrieves unit preference from user config
- Handler layer extracts user unit preference via middleware, passes to DTO conversion
- Frontend can also do display-side conversion based on user `unit_system` preference

## Appendix A: Architecture Decision Records (ADR)

### ADR-001: Modular Monolith over Microservices

| Item | Content |
|------|---------|
| **Context** | Medium project size, small team |
| **Decision** | Modular monolith with domain-based internal packaging, no separate microservices |
| **Rationale** | Lower ops complexity; SQLite single-DB doesn't support microservices; can extract along module boundaries later if needed |
| **Consequences** | Need strict module boundary design to prevent cross-module coupling |

### ADR-002: SQLite over PostgreSQL

| Item | Content |
|------|---------|
| **Context** | Data volume ~hundreds of thousands of rows, 20+ concurrent users |
| **Decision** | SQLite WAL mode with connection pool and write serialization |
| **Rationale** | Zero config, file-level deployment, adequate for this scale; SQLite can handle millions of reads and moderate write concurrency |
| **Consequences** | Can migrate to PostgreSQL later if needed (SQLx has good compatibility with both) |

### ADR-003: Imperial Units as Internal Storage Standard

| Item | Content |
|------|---------|
| **Context** | API 5CT standard uses imperial; domestic users prefer metric |
| **Decision** | DB stores imperial values uniformly; API layer converts based on user preference |
| **Rationale** | Avoids precision loss (repeated metric↔imperial conversion accumulates error); stays aligned with standard |
| **Consequences** | All internal computation in imperial; unit conversion on API input/output |

### ADR-004: Separate Tables for Seamless and Screen Pipe

| Item | Content |
|------|---------|
| **Context** | Both pipe types share common attributes, but screen pipe has base pipe params, filtration precision, etc. |
| **Decision** | Two independent tables, no shared schema |
| **Rationale** | Significant field differences (screen pipe has unique fields and its base pipe is itself a seamless pipe parameter); combined queries are rare; separate tables are cleaner |
| **Consequences** | Cross-type search requires `UNION` queries or two independent queries merged |
