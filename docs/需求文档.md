## 1. Why We're Building This

### 1.1 The Problem

Seamless pipe (Casing / Tubing) and screen pipe are the backbone of oil & gas well construction. Both follow **API 5CT**. Problem is, there's no decent integrated system to track these things through their whole lifecycle — from purchase order, to warehouse arrival, to sitting in stock, to shipping out the door, to quality traceability. It's all spreadsheets and gut feelings.

### 1.2 What We're Solving

Build a web-based management system (Rust backend) that covers the full **procurement + sales + inventory + quality** lifecycle for seamless pipes and screen pipes. One system to rule them all, from the mill to the wellhead.

## 3. Features

### 3.1 Pipe Master Data (P0 — Ship Now or Don't Ship)

**FR-PIPE-001: Pipe Records**
- **What**: CRUD for both seamless and screen pipe types
- **Seamless pipe fields**:
  - Pipe number (unique — no duplicates, we're not animals)
  - Product type: Casing / Tubing
  - Grade: H40, J55, K55, N80, L80, C90, T95, P110, Q125, etc.
  - Specs: outer diameter (OD), wall thickness (WT), length, weight per unit
  - End type: Short Round (SC), Long Round (LC), Buttress (BC), Extreme Line, etc.
  - Coupling specs: type, OD, length
  - Heat Number / Lot Number
  - Body serial number
- **Screen pipe fields**:
  - Screen type: Wire-wrapped, Slotted, Punched, Metal mesh, etc.
  - Base pipe specs: OD, WT, grade (the base pipe is itself a seamless pipe)
  - Slot width / hole diameter
  - Filtration rating (e.g. 150μm, 250μm)
  - Screen length and connection type
- **Common fields**: manufacturer, production date, cert number, attachments (scanned certs, etc.)
- **Acceptance criteria**:
  - Create / update / delete pipe records
  - Pipe numbers are globally unique — no collisions
  - Search by any combo of fields
  - Screen and seamless pipes live in separate types but are queryable from a unified view

### 3.2 Inventory Management (P0 — Ship Now or Don't Ship)

**FR-INV-001: Inbound**
- Support multiple inbound types: purchase receipt, production completion, return, etc.
- Record: inbound order number, date, pipe details (numbers / qty / batches), supplier/source, operator
- Auto-update stock counts — because nobody wants to reconcile manually

**FR-INV-002: Outbound**
- Support multiple outbound types: sales, internal use, transfer, etc.
- Record: outbound order number, date, customer/destination, pipe details, operator
- Support batch and per-piece outbound (pick by exact pipe number)

**FR-INV-003: Stock Query & Counting**
- Real-time stock lookup: filter by pipe type, grade, spec, location, whatever
- Movement history: every single pipe's inbound/outbound trail
- Physical counting: generate count sheets, enter count data, produce discrepancy reports

**FR-INV-004: Locations**
- Hierarchical location management: zone → shelf → level
- Pipes are tied to locations, support location transfers

### 3.3 Quality Management (P1 — Should Have)

**FR-QA-001: QC Records**
- Link inspection certs / mill certs to pipes
- Record: test items, results, date, agency, inspector
- Upload attachments (PDFs, images)

**FR-QA-002: Quality Traceability**
- Trace by heat number back to production batch
- Trace by pipe number to every quality record for that single pipe
- Reference API 5CT standard mechanical properties and chemical composition data — for comparison and sanity checking

### 3.4 Procurement & Sales (P1 — Should Have)

**FR-PUR-001: Purchasing**
- Create, approve, and track purchase orders
- Supplier management (name, contact, qualifications, etc.)
- Inbound links to PO — when goods arrive, they're tied to the order

**FR-SALE-001: Sales**
- Create, approve, and track sales orders
- Customer management
- Outbound decrements stock automatically
- Show Available-to-Promise (ATP) — so sales doesn't oversell what we don't have

**FR-CONTRACT-001: Contracts**
- Basic contract info for both buy-side and sell-side
- Contracts are linked to orders

### 3.5 Data Import/Export (P1 — Should Have)

**FR-IO-001: Import**
- Batch import pipe data from Excel / CSV
- Validate format and required fields on import
- Report what worked and what didn't (success/failure rows with reasons)

**FR-IO-002: Export**
- Export query results to Excel / CSV
- Pre-built reports: stock summary, inbound/outbound details, quality reports

### 3.6 Search & Filtering (P0 — Ship Now or Don't Ship)

**FR-SEARCH-001: Multi-Dimension Search**
- Combine filters on pipe number, type, grade, specs (OD/WT), status, location — whatever combo you want
- Fuzzy search (partial pipe numbers, grade names, etc.)
- Paginated results — because nobody wants to scroll through 10,000 pipes

### 3.7 Reports & Analytics (P2 — Nice to Have)

**FR-RPT-001: Inventory Reports**
- Current stock overview (grouped by type / grade / spec)
- Monthly / quarterly stock movement reports

**FR-RPT-002: Business Reports**
- Inbound/outbound charts
- Inventory turnover analysis
- Purchase / sales trend analysis

### 3.8 Label Printing (P2 — Nice to Have)

**FR-LABEL-001: Barcode / QR Label Printing**
- Generate barcode or QR labels from pipe data
- Batch printing
- Configurable label templates (what fields to show, layout)

### 3.9 Full Traceability (P0 — Ship Now or Don't Ship)

**FR-TRACE-001: Lifecycle Traceability**
- Log every operation on every pipe, from inbound to outbound
- Log every edit: who changed what field and when
- View the complete lifecycle by pipe number

### 3.10 System Admin (P1 — Should Have)

**FR-SYS-001: Users & Permissions**
- User registration / management
- Role-based access (RBAC)
  - Warehouse, QC, Sales/Procurement, Admin
- Different roles see different menus and buttons

**FR-SYS-002: Operation Logs**
- Log user logins, data modifications, and other critical actions
- Logs are searchable and exportable

## 4. Non-Functional Stuff

### 4.1 Performance

| Metric | Target |
|--------|--------|
| Single-page query response | ≤ 2 seconds (under 100k records) |
| Data import | 100k records in ≤ 60 seconds |
| Concurrent users | ≥ 20 simultaneous users |
| Uptime | 99.5% (≤ 44 hours downtime/year) |

### 4.2 Data Volume

- Pipe master data: hundreds of thousands (≥ 100k records)
- Inventory movement logs: millions of records
- Storage: SQLite (indexes need to be sane for this scale)

> **Heads up**: SQLite is fine for single-instance / low-concurrency setups. For a multi-user web app with write contention, make sure WAL mode is on and the connection pool is configured properly. If concurrency needs grow, PostgreSQL migration is the exit strategy.

### 4.3 Internationalization & Units

- **UI languages**: Chinese and English, toggleable
- **Unit systems**: metric (mm, kg/m, m) and imperial (in, lb/ft, ft), toggleable
- Input can switch between unit systems; internal storage uses metric (or includes unit markers)

### 4.4 Security

- Passwords hashed with Argon2id (not bcrypt, not plaintext — obviously)
- Destructive operations (deletes, critical edits) need confirmation
- API auth via JWT
- HTTPS in production (duh)

### 4.5 Maintainability

- Rust's type system catches a ton of bugs at compile time — fewer runtime surprises
- Modular architecture with clear package boundaries
- Backend exposes RESTful APIs, frontend consumes them via HTTP
- API docs via OpenAPI / Swagger

### 4.6 Stack Choices

| Layer | Pick | Why |
|-------|------|-----|
| **Backend** | Rust + Axum + SQLx | Axum is the de facto standard async web framework in Rust land. SQLx checks SQL at compile time — love it or hate it, it catches mistakes. |
| **Database** | SQLite (WAL mode) | Zero config, file-level DB. Good enough for this scale. |
| **Frontend** | React 19 + Ant Design 5 + TanStack Query 5 + Zustand 5 | React is mature. Ant Design gives us enterprise-grade components out of the box. TanStack Query handles server state caching without us reinventing the wheel. Zustand for client-side state. |
| **API** | JSON REST | Standard, boring, works with everything. |

## 5. API 5CT Standards Reference

> API 5CT (Casing and Tubing Specification) is *the* standard for oil & country tubular goods. Here's the reference data that drives our field design.

### 5.1 Common Grades

| Grade Group | Grade | Use Case | Vibe |
|------------|-------|----------|------|
| **H** | H40 | Casing/Tubing | Lowest strength, non-critical wells |
| **J/K** | J55, K55 | Casing/Tubing | Medium strength, medium-depth wells |
| **N** | N80 | Casing/Tubing | Higher strength. Comes in N80-1 and N80-Q (different heat treatment) |
| **L** | L80 | Casing/Tubing | Corrosion resistant. Has Cr. For H₂S environments. |
| **C** | C90, C95 | Casing/Tubing | Corrosion resistant. Sour service. |
| **T** | T95 | Casing | High collapse resistance. Also for sour service. |
| **P** | P110 | Casing/Tubing | High strength. Deep wells. |
| **Q** | Q125 | Casing | Ultra-high strength. Ultra-deep wells. |

### 5.2 End Types

- **SC** (Short Round Thread) — shallow wells, basic connections
- **LC** (Long Round Thread) — stronger than SC
- **BC** (Buttress Thread) — high connection strength
- **X** (Extreme Line) — special cases, flush joint connections

### 5.3 Units

API 5CT is imperial by nature:

| Parameter | Imperial | Metric |
|-----------|----------|--------|
| Outer Diameter (OD) | inch (in) | millimeters (mm) |
| Wall Thickness (WT) | inch (in) | millimeters (mm) |
| Length | foot (ft) | meters (m) |
| Unit Weight | lb/ft | kg/m |
| Yield Strength | psi | MPa |

### 5.4 Reference Sizes

| OD (in) | WT (in) | Unit Wt (lb/ft) | Common Grades |
|---------|---------|-----------------|---------------|
| 4½ | 0.250 ~ 0.337 | 11.60 ~ 15.10 | J55, N80, L80 |
| 5½ | 0.304 ~ 0.415 | 17.00 ~ 23.00 | J55, N80, L80, P110 |
| 7 | 0.317 ~ 0.582 | 23.00 ~ 41.00 | J55, N80, L80, P110 |
| 9⅝ | 0.395 ~ 0.595 | 40.00 ~ 59.20 | J55, N80, L80, P110 |
| 13⅜ | 0.514 ~ 0.672 | 72.00 ~ 92.50 | H40, J55, K55 |

## 6. Data Model (First Pass)

### 6.1 Core Entity Relationships

```
                      ┌──────────────────────┐
                      │   SeamlessPipe        │  ← Seamless pipe (its own table)
                       │   (Seamless Pipe)       │
                      └────────┬─────────────┘
                               │
Supplier ──→ PurchaseOrder ──→ InboundRecord ──→ SeamlessPipe/ScreenPipe ──→ OutboundRecord ──→ Customer
                     ↑                                                          │
                Contract                                                   SalesOrder
                     │                                                          ↑
                     └──────────────────────────────────────────────────────────┘

                      ┌──────────────────────┐
                      │   ScreenPipe           │  ← Screen pipe (its own table too)
                       │   (Screen Pipe)         │
                      └──────────────────────┘
```

> SeamlessPipe and ScreenPipe are **two separate tables**. Each has its own full set of fields. No shared table, no single-table inheritance nonsense.

### 6.2 Main Entities

| Entity | What It Is | Key Fields |
|--------|-----------|------------|
| **SeamlessPipe** | Seamless pipe (casing/tubing) | id, pipe_number, pipe_type(casing/tubing), grade, od, wt, length, weight, end_type, coupling_type, coupling_od, coupling_length, heat_number, serial_number, manufacturer, production_date, cert_number, location_id, status, created_at, updated_at |
| **ScreenPipe** | Screen pipe | id, pipe_number, screen_type(wire-wrapped/slotted/punched), slot_size, filtration_grade, base_od, base_wt, base_grade, base_end_type, length, weight, heat_number, serial_number, manufacturer, production_date, cert_number, location_id, status, created_at, updated_at |
| **Location** | Warehouse bin location | id, zone_code, shelf_code, level_code, description |
| **Supplier** | Vendor that sells us stuff | id, name, contact, phone, address, qual_cert |
| **Customer** | Person/company buying from us | id, name, contact, phone, address |
| **PurchaseOrder** | PO header | id, order_no, supplier_id, order_date, status, total_amount |
| **SalesOrder** | SO header | id, order_no, customer_id, order_date, status, total_amount |
| **InboundRecord** | Receiving record | id, record_no, type, pipe_type(seamless/screen), pipe_id, quantity, date, order_id, operator, remark |
| **OutboundRecord** | Shipping record | id, record_no, type, pipe_type(seamless/screen), pipe_id, quantity, date, order_id, operator, remark |
| **InventoryLog** | Movement audit trail | id, pipe_type(seamless/screen), pipe_id, change_type, quantity_before, quantity_after, operation, operator, timestamp |
| **QualityCert** | Inspection certificate | id, pipe_type(seamless/screen), pipe_id, cert_no, inspect_date, inspector, agency, file_url, result, remark |
| **User** | System user | id, username, password_hash, role, name, email, language_pref, unit_system |
| **OperationLog** | Audit log | id, user_id, action, target_type, target_id, detail, ip_address, timestamp |
